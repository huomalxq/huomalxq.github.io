# Git 仓库学习记录维护流程

## 背景

这个个人站点本身就是一个 GitHub Pages 仓库，学习记录也放在同一个仓库中维护。每次新增笔记时，需要同时关注三个动作：写内容、更新导航、验证构建。

如果只新增 Markdown 文件而不更新 `mkdocs.yml`，页面可能能被构建出来，但不会出现在顶部导航或侧边目录中；如果没有本地构建验证，就容易把格式、链接或缩进问题留到 GitHub Actions 阶段才暴露。

## 过程

一次比较稳的维护流程可以按下面顺序执行：

```bash
git status --short
```

先确认当前工作区是否干净。如果已经有未提交内容，需要区分哪些是本次学习记录要改的，哪些是之前留下的变更。

```bash
mkdir -p docs/study/engineering
```

根据主题选择分类目录。工程实践类内容放在 `docs/study/engineering/`，AI、Agent 或模型相关内容放在 `docs/study/ai/`。

```bash
mkdocs build
```

新增或修改页面后，用本地构建检查导航、Markdown 语法和站内链接。这个仓库的发布流程会在推送到 `main` 分支后由 GitHub Actions 自动构建，因此本地构建相当于提前做一次小型预检。

## 结论

学习记录不是单独存在的文本文件，而是站点结构的一部分。比较适合保留的习惯是：

- 新增页面后同步更新 `mkdocs.yml`。
- 分类页只负责做入口和说明，不堆太多正文。
- 每次提交前运行一次 `mkdocs build`。
- 提交前用 `git diff` 快速复查是否只改了预期文件。

## 后续

后续可以考虑补一篇关于 GitHub Actions 部署失败排查的记录，把常见问题例如依赖安装失败、导航路径错误、Pages 权限配置放在一起整理。
