# 开放原子大赛 Teco-RAG 项目总结

## 项目背景

Teco-RAG 项目来源于开放原子大赛的大模型开发任务，目标是围绕检索增强生成（Retrieval Augmented Generation, RAG）构建面向知识问答场景的解决方案。

传统大语言模型在回答专业知识、私有知识或长尾问题时，容易出现知识缺失、幻觉和答案不可追溯等问题。RAG 的核心思路是先从外部知识库中检索相关文档片段，再把这些片段作为上下文交给大模型生成答案，从而提升准确性、可解释性和知识可更新性。

我负责了项目从方案设计到实验落地的完整流程，包括 RAG 框架搭建、向量数据库接入、文档切分策略设计、Embedding 与 Rerank 模型选型、检索链路优化、参数调优和最终方案总结。

## Baseline 方案

项目的 baseline 系统包含两个阶段：

1. 索引阶段：清洗原始知识文档，切分为 chunk，向量化后写入向量数据库。
2. 检索生成阶段：根据用户问题检索相关知识片段，经 Rerank 筛选后拼接到 prompt，交给大模型生成答案。

基础配置如下：

- 向量数据库：Milvus
- Embedding 模型：`BAAI/bge-large-zh`
- Rerank 模型：`BAAI/bge-reranker-large`
- 文档切分器：`ChineseRecursiveTextSplitter`
- 文档切分参数：`chunk_size = 500`，`chunk_overlap = 50`
- 生成模型：`Qwen-7B-Chat`

## 我的工作

- 搭建 RAG 基线系统，串联 Milvus、Embedding、Rerank 和 Qwen-7B-Chat。
- 设计文档切分实验，对比不同 chunk size 和 chunk overlap 对检索效果的影响。
- 对比多种中文 Embedding 模型，包括 `bge-large-zh`、`bce-embedding-base_v1`、`Conan-embedding-v1`、`Chunxin-Embedding` 等。
- 引入多查询、HyDE、Query 重写和 Rerank 再排序，优化检索链路。
- 调整 Top-k 和 Top-n，平衡召回覆盖与最终上下文质量。
- 尝试 BM25 稀疏检索，用于补充稠密向量检索在精确词匹配上的不足。
- 负责实验结果分析、方案迭代和答辩材料整理。

## 索引优化

文档切分会直接影响后续向量化、召回和生成效果。我对多组切分参数进行了实验：

- `1536-200`
- `1024-200`
- `700-200`
- `700-100`
- `300-50`

过短的 chunk 容易导致上下文信息不足，过长的 chunk 又会引入噪声，影响召回和重排。因此需要在语义完整性、检索精度和生成上下文长度之间取得平衡。

常见切分方法包括：

- 固定长度切分：实现简单，但可能切断完整句子或表格。
- 滑动窗口切分：通过 overlap 保留相邻 chunk 的连续性。
- 递归切分：按标题、段落、标点逐级切分，更适合中文文本。
- 按章节切分：适合技术文档、论文、规章制度和产品说明书。
- 语义切分：根据相邻句段语义相似度决定切分位置，效果更好但成本更高。

在 Embedding 模型方面，我对比了多种中文模型。其中 `bce-embedding-base_v1` 相比 baseline 约有 6% 提升，说明模型选型对中文语义检索效果影响明显。

## 检索优化

检索阶段主要解决“如何更准确地找到相关知识”的问题。我采用了以下策略：

- 多查询：让生成模型扩展出 3 个语义相近的问题，分别检索，扩大召回范围。
- HyDE：先让模型生成假设性答案，再用伪答案检索，使检索语义更接近目标答案，实验中约带来 3% 提升。
- Query 重写：当原始问题表述不清时，先改写问题再检索和生成。
- Rerank 再排序：用重排模型筛掉无关片段，减少噪声进入最终 prompt。
- Top-k / Top-n 调优：先召回更多候选，再只保留更精准的少量片段。

最终我将 Top-k 设置为 32，Top-n 设置为 3。相比默认 5-5 配置，这种“宽召回 + 严筛选”的策略约提升 13%。

## BM25 稀疏检索

除稠密向量检索外，我还尝试了 BM25 稀疏检索。BM25 基于词频、逆文档频率和文档长度计算 query 与文档之间的相关性。相比稠密向量检索，BM25 在关键词匹配明确的问题上速度更快，也能补充向量检索对精确词匹配的不足。

BM25 的基本公式为：

$$
\operatorname{score}(D, Q) =
\sum_{q_i \in Q}
\operatorname{IDF}(q_i)
\cdot
\frac{f(q_i, D) \cdot (k_1 + 1)}
{f(q_i, D) + k_1 \cdot \left(1 - b + b \cdot \frac{|D|}{\operatorname{avgdl}}\right)}
$$

其中，$D$ 表示文档或 chunk，$Q$ 表示用户查询，$q_i$ 表示查询中的词，$f(q_i, D)$ 表示词在文档中的出现次数，$|D|$ 表示文档长度，$\operatorname{avgdl}$ 表示平均文档长度。

BM25 的核心逻辑是：如果一个文档包含 query 的关键词，这些关键词又比较稀有，并且文档长度没有带来不公平优势，那么该文档会获得更高相关性得分。

## 问题与解决

### 文档切分粒度不好控制

初始方案中 `chunk_size = 500`、`chunk_overlap = 50` 虽然简单，但部分问题会出现语义不完整或上下文噪声过多的情况。

我通过多组 chunk size 和 overlap 实验，综合比较召回效果和生成质量，选择更适合任务数据的切分策略。

### 单一 query 召回不足

用户问题表达方式可能和知识库原文不一致，直接使用原始 query 检索容易漏掉语义相关内容。

我引入多查询和 HyDE，先扩展查询表达，再进行多路检索，提高召回覆盖面。

### 召回文档存在噪声

向量数据库返回的 Top-k 结果不一定都适合进入 prompt。如果全部输入模型，容易造成上下文冗余和答案偏移。

我使用 Rerank 模型对候选文档排序，并采用 Top-k 32、Top-n 3 的配置，既保证召回范围，又控制最终输入质量。

### 中文 Embedding 效果差异明显

不同 Embedding 模型在中文语义表达、领域文本适配和向量空间质量上差异较大。

我对比多个模型，并根据实际评测选择效果更好的 `bce-embedding-base_v1`。

### 检索速度与精度需要平衡

稠密向量检索语义能力强，但并不总是最快或最适合精确关键词匹配。

我尝试 BM25 作为稀疏检索补充，并结合 Rerank 排序提升检索稳定性。

## 项目收获

这个项目让我完整实践了一个 RAG 系统从 baseline 到优化的过程。RAG 的效果并不只取决于生成模型，索引质量、切分策略、Embedding 模型、召回数量、重排质量和 prompt 组织方式都会影响最终答案。

一个高质量 RAG 系统需要同时做好“检索准确性”和“生成可用性”，不能只堆叠模型。
