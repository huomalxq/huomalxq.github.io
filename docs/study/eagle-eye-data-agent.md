# Eagle Eye 数据分析 Agent 后端项目复盘

## 项目背景

Eagle Eye 是一个面向企业内部数据分析场景的智能问数后端服务。业务用户不一定了解底层数据库结构，也不一定会写 SQL，但需要快速查询设备、场地、医院、扫描、日志等业务数据，并拿到可执行 SQL、查询结果和初步分析。

项目目标是把“自然语言提问、理解业务意图、检索相关 schema、生成 SQL、执行校验、结果分析、图表配置生成”串成一条后端 Agent 链路。它不是单纯的聊天机器人，而是面向真实企业数据平台的智能数据分析助手。

## 我的职责

我负责项目后端的整体设计与实现，主要工作包括：

- 使用 FastAPI 搭建后端 API，提供自然语言生成 SQL、SQL 执行和结果分析接口。
- 使用 LangGraph 设计 Agent 工作流，将意图识别、RAG 检索、SQL 生成、SQL 校验和失败重试拆成可观测节点。
- 使用 Milvus 存储表结构、字段描述、业务实体和业务字典，并通过 Ollama embedding 生成向量。
- 接入 vLLM 的 OpenAI 兼容接口，支持意图识别、关键词提取、SQL 生成和分析结论生成。
- 接入 Vertica 查询业务数据，接入 PostgreSQL 保存问答历史和元数据。
- 对 SQL 输出做清洗、LIMIT 控制、小样本执行校验、失败原因反馈和自动重试。
- 将 SQL 查询结果转换为 DataFrame，生成数据 profile、分析摘要和 Chart.js 图表配置。
- 编写启动脚本、知识库重建脚本、RAG 测试脚本和完整 API 测试脚本。

## 系统架构

项目整体可以拆成五层：

1. API 接入层：FastAPI 对外提供 `/workflow_stream`、`/execute-and-analyze` 等接口。
2. Agent 编排层：LangGraph 管理多步骤推理流程，每个节点负责一个明确任务。
3. LLM 能力层：vLLM 提供本地大模型推理服务，兼容 OpenAI 调用方式。
4. RAG 知识层：Ollama embedding + Milvus 存储和召回 schema、实体、字典知识。
5. 数据访问层：Vertica 查询业务数据，PostgreSQL 保存历史记录和平台侧数据。

核心链路如下：

```text
用户问题
  -> FastAPI 接口
  -> LangGraph 工作流
  -> 意图识别与关键词提取
  -> RAG 检索表结构、字段、实体、字典
  -> 组装 schema prompt
  -> LLM 生成 SQL
  -> 小样本 SQL 执行校验
  -> 失败反馈重试 / 成功返回 SQL
  -> 执行完整查询
  -> 生成分析结论和图表配置
  -> 前端展示
```

## 关键问题与处理方式

### 业务问题很难直接映射到数据表

用户通常会用业务语言提问，例如“某个场地扫描情况”或“医院设备状态”，但数据库表名和字段名通常是英文缩写。直接让模型写 SQL，很容易选错表或编造字段。

我的处理方式是先让 LLM 从用户问题中提取关键词，再用原始问题和关键词分别检索实体、表结构和业务字典。召回后把表 DDL、字段描述、业务实体、字典说明和表关系一起注入 prompt，并记录召回结果，方便后续评估 RAG 质量。

### SQL 生成不稳定

大模型生成的 SQL 可能夹杂说明文字、缺少 LIMIT、引用不存在字段，或者不符合 Vertica 方言。

我在后端增加了多层约束：prompt 限制只生成 SELECT，正则提取 SQL 片段，自动添加或收紧 LIMIT，并先做小样本查询校验。执行失败时，将数据库错误写回工作流状态，触发最多三次重试。

### RAG 召回结果有噪声

单一路向量检索很难覆盖业务实体、字段别名和数据字典之间的多种表达。

我将知识库拆成实体集合、表结构集合和字典集合，通过多路召回补足上下文。高置信度字典结果会提高优先级，表关系映射用于补充 join 条件。召回过程会输出到日志和 Excel，方便离线分析。

### 数据库查询成本不可控

企业数据表数据量大，如果 SQL 范围过宽，接口可能长时间等待，甚至影响服务稳定性。

执行层加入了超时控制。SQL 校验阶段只查询少量样本，最终返回给前端的数据会做预览截断，并记录 SQL 执行耗时，便于定位慢查询。

### 模型输出的图表配置格式不稳定

LLM 生成图表 spec 时可能输出 markdown、额外解释或非法 JSON，也可能引用不存在的字段。

我在 prompt 中要求固定输出 `[ANALYSIS]` 和 `<visual_spec>` 标签，后端用正则提取 JSON 数组，做容错解析和字段名大小写匹配。图表生成失败时降级为空图表，保证主接口不崩溃。

## 项目亮点

- 将企业数据仓库问数流程封装成可执行的数据分析 Agent，而不是普通问答机器人。
- 使用 LangGraph 将复杂推理拆成可观测、可重试的节点。
- 使用 Milvus + embedding 构建面向 schema、业务实体和字典的 RAG 检索。
- SQL 生成后先执行校验，再返回或进入错误反馈重试，提高可用性。
- SQL 查询结果可以继续生成分析摘要和图表配置，形成完整闭环。
- 通过日志、测试脚本和召回记录持续评估 SQL 成功率、召回质量和各阶段耗时。

## 可量化指标

后续评估时可以重点跟踪这些指标：

- RAG Top-k 是否召回目标表。
- SQL 一次生成成功率和重试后成功率。
- RAG、SQL 生成、SQL 执行、结果分析的阶段耗时。
- SQL 失败原因占比，例如字段不存在、表选错、语法错误、查询超时。
- 图表配置可解析率和前端渲染成功率。

## 面试表达

一分钟版本：

> 我负责过一个企业内部智能数据分析 Agent 的完整后端。项目目标是让业务用户通过自然语言查询复杂数据库，系统自动完成意图识别、RAG 检索相关表结构、生成 SQL、执行校验、失败重试，并进一步对查询结果生成分析结论和图表配置。技术上使用 FastAPI 提供接口，LangGraph 编排 Agent 工作流，vLLM 提供大模型服务，Ollama 生成 embedding，Milvus 做 schema 和业务知识检索，Vertica/PostgreSQL 负责数据查询和历史记录。项目难点主要在复杂 schema 下提高 SQL 生成稳定性，所以我把流程拆成可观测、可重试的多个节点，并通过 RAG、多路召回、SQL 小样本校验、错误反馈重试和日志评估来提升整体可靠性。

## 后续优化

- 将明文配置迁移到环境变量或密钥管理系统。
- 将项目中的绝对路径改为基于项目根目录的相对路径。
- 使用 Redis checkpoint 替代内存 checkpoint，增强服务重启后的状态恢复能力。
- 引入 SQL parser 做更严格的安全校验。
- 建立标准评测集，持续评估 RAG 召回率和 SQL 成功率。
- 对常见问题做缓存，降低模型调用成本。
