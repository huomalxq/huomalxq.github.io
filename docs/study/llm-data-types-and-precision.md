# 大模型数据格式与混合精度知识点总结

来源：[Datawhale LLM-Algo-LeetCode：Data Types and Precision](https://datawhalechina.github.io/llm-algo-leetcode/01_Hardware_Math_and_Systems/01_Data_Types_and_Precision.html)

## 核心结论

大模型里的数据格式选择，本质是在 **显存占用、数值范围、表示精度、硬件吞吐** 之间取平衡。

训练阶段更怕数值溢出，所以现代大模型预训练更偏向 BF16；推理阶段更关注吞吐和显存，所以常见 FP16、INT8、INT4 等压缩格式；新一代硬件则进一步把 FP8 推向训练和推理。

## 1. 常见数据格式占用

| 格式 | 位宽 | 每个参数占用 | 常见用途 |
| --- | ---: | ---: | --- |
| FP32 | 32 bit | 4 Byte | 高精度训练、调试、优化器主权重 |
| FP16 | 16 bit | 2 Byte | 推理、早期混合精度训练 |
| BF16 | 16 bit | 2 Byte | 现代大模型预训练主流格式 |
| INT8 | 8 bit | 1 Byte | 推理量化 |
| INT4 | 4 bit | 0.5 Byte | 极限压缩量化，如 AWQ、GPTQ |

静态权重显存估算可以先用：

```text
模型参数量 × 单个参数字节数
```

例如 7B 模型用 FP16 或 BF16 加载，仅权重大约需要：

```text
7 × 10^9 × 2 Byte ≈ 14 GB
```

这只是模型权重，不包含 KV Cache、激活值、梯度、优化器状态等额外显存。

## 2. 浮点数的底层结构

浮点数通常由三部分组成：

- 符号位：表示正负。
- 指数位：决定可表示数值范围。
- 尾数位：决定小数表示精度。

可以记成一句话：

```text
指数位管范围，尾数位管精度。
```

FP16 和 BF16 都是 16 bit，但位分配完全不同：

| 格式 | 符号位 | 指数位 | 尾数位 | 特点 |
| --- | ---: | ---: | ---: | --- |
| FP16 | 1 | 5 | 10 | 小数更精细，但动态范围窄，最大约 65504 |
| BF16 | 1 | 8 | 7 | 继承 FP32 的指数范围，更不容易溢出 |

FP16 把更多 bit 给了尾数，因此单个数的小数精度更好；BF16 把更多 bit 给了指数，因此数值范围接近 FP32，更适合深层网络训练。

## 3. 为什么大模型训练偏向 BF16

FP16 在大模型训练里的核心问题是动态范围太窄。Attention logits、未归一化激活、梯度等值一旦超过 FP16 可表示上限，就可能产生 `NaN`，训练会直接崩溃。

FP16 的另一个问题是小梯度下溢。早期会用 Loss Scaling 缓解：先放大 loss 和梯度，再在更新时缩回来。但它主要缓解小梯度下溢，不能根治前向过程里的上溢，而且需要动态调缩放因子，工程复杂度高。

BF16 的优势在于：

- 动态范围接近 FP32，训练中更难上溢。
- 不需要复杂的 Loss Scaling。
- A100 之后的 Tensor Core 原生支持 BF16，性能接近 FP16。
- 神经网络对少量尾数精度损失比较鲁棒，但对 `NaN` 非常脆弱。

因此，BF16 用少量精度代价换来了大幅数值稳定性，成为 LLaMA、GPT-3、PaLM 等大模型预训练中更常见的选择。

## 4. 混合精度训练为什么保留 FP32 主权重

混合精度训练中，前向和反向可以用 16-bit 跑，以减少显存并利用 Tensor Core 加速。但参数更新这一步仍然需要更高精度：

```text
W_new = W_old - Learning_Rate × Gradient
```

训练后期学习率很小，梯度也可能很小，二者乘积得到的更新量非常微弱。如果权重本身只用 16-bit 保存，微小更新可能因为尾数精度不足而被舍入掉，表现为“模型吃不到更新”。

所以优化器通常保留一份 FP32 Master Weights：

1. 16-bit 权重用于前向和反向计算。
2. 梯度产生后，在 FP32 主权重上做参数更新。
3. 更新后的 FP32 权重再转换成 16-bit，用于下一轮计算。

这也是全参训练显存远高于“模型权重大小”的重要原因之一，因为还要保存梯度、优化器状态和 FP32 主权重。

## 5. A100 的关键精度升级

A100 让大模型训练变得更稳定、更高效，关键在两个精度能力。

### 原生 BF16 Tensor Core

V100 时代 Tensor Core 主要支持 FP16，训练大模型时更容易遇到 FP16 溢出问题。A100 的 Tensor Core 原生支持 BF16 乘加，使 BF16 训练既稳定又高吞吐。

### TF32

TF32 是 NVIDIA 为 FP32 矩阵乘法设计的加速格式。它在计算内部使用类似“FP32 指数位 + FP16 尾数位”的组合：

- 指数位保持 8 位，动态范围接近 FP32。
- 尾数位约 10 位，精度低于 FP32，但吞吐更高。
- 显存中仍按 FP32 存储，对用户代码侵入较小。

在 PyTorch 中，A100 及更新架构可以通过 `torch.backends.cuda.matmul.allow_tf32 = True` 让 FP32 矩阵乘自动走 TF32 Tensor Core 加速。

## 6. H100 的 FP8 思路

H100 引入原生 FP8，进一步压缩数据宽度、提升吞吐。由于 8 bit 空间太小，FP8 分成两种变体服务不同阶段：

| FP8 格式 | 位分配 | 侧重点 | 典型用途 |
| --- | --- | --- | --- |
| E4M3 | 4 位指数 + 3 位尾数 | 更重视精度 | 前向传播、激活值 |
| E5M2 | 5 位指数 + 2 位尾数 | 更重视动态范围 | 反向传播、梯度 |

前向激活通常数值分布更集中，对精度更敏感，所以适合 E4M3；反向梯度跨度更大，更容易下溢，因此适合指数位更多的 E5M2。

## 7. 面试速记

- 数据格式显存估算：`参数量 × 每参数字节数`。
- FP16 和 BF16 都是 2 Byte，但 FP16 尾数更多、范围更窄；BF16 指数更多、范围接近 FP32。
- 现代大模型训练常用 BF16，因为它显著降低上溢和 `NaN` 风险。
- Loss Scaling 主要缓解 FP16 梯度下溢，无法根治前向上溢。
- AMP 训练仍保留 FP32 Master Weights，是为了让微小梯度更新能被准确累加。
- A100 的重要升级是原生 BF16 Tensor Core 和 TF32。
- H100 的 FP8 分 E4M3 与 E5M2，分别偏向前向精度和反向动态范围。

## 8. 一句话总结

大模型精度体系可以按时代理解：FP32 稳但慢，FP16 快但容易溢出，BF16 在训练中兼顾稳定与速度，TF32 给 FP32 代码自动加速，FP8 则代表更激进的低精度训练和推理方向。
